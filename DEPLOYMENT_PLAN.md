# DEPLOYMENT_PLAN.md

## Назначение

Документ описывает предварительный план будущего развёртывания `permit-system` на сервере организации. План нужен для обсуждения с ИТ-службой, подготовки тестового окружения и фиксации инфраструктурных рисков.

> Важно: это план развёртывания и список вопросов. Он не меняет код приложения, модели, workflow или бизнес-логику.

## 1. Предполагаемая схема развёртывания

### Django app

- Основное приложение — Django-проект `permit_system`.
- Запуск возможен через Docker Compose или через Python virtualenv/systemd, если Docker недоступен.
- Приложение должно запускаться с `DEBUG=0` в тестовом/серверном окружении.
- Внешний доступ должен идти через доменное имя или IP, указанные в `ALLOWED_HOSTS`.
- Reverse proxy/HTTPS можно добавить отдельным инфраструктурным слоем после первичного тестового развёртывания.

### PostgreSQL

- Основная серверная база данных — PostgreSQL.
- Подключение приложения желательно задавать через `DATABASE_URL`.
- Для локальной offline-проверки без PostgreSQL используется SQLite, но для серверного окружения нужен PostgreSQL.
- Нужно заранее определить место хранения данных PostgreSQL и регламент резервного копирования.

### Media storage

- `MEDIA_ROOT` хранит загруженные DOCX-шаблоны и сгенерированные документы.
- Media-файлы должны храниться на постоянном диске/volume, а не внутри эфемерного контейнера.
- Нужно определить права доступа к каталогу media и политику резервного копирования.
- В будущем может понадобиться отдельное файловое хранилище или сетевой диск.

### Static files

- `STATIC_ROOT` хранит собранную статику Django после `collectstatic`.
- На тестовом этапе статика может обслуживаться приложением/контейнером, если это допустимо для внутреннего стенда.
- Для production-подхода лучше отдавать static через reverse proxy или отдельный web-сервер.

### Optional LibreOffice/soffice for PDF

- PDF-конвертация является опциональной.
- По умолчанию `PDF_CONVERTER_ENABLED=0`, поэтому отсутствие LibreOffice/`soffice` не должно ломать DOCX-генерацию.
- Для реального PDF-экспорта нужно установить LibreOffice/`soffice` или согласованный аналог, например Р7-Офис, если он поддерживает нужный режим конвертации.
- Путь к исполняемому файлу задаётся через `SOFFICE_PATH`.

### Backup

- Минимальный backup должен включать:
  - PostgreSQL database dump;
  - каталог `MEDIA_ROOT` с шаблонами и сгенерированными документами;
  - актуальный `.env` или безопасно сохранённые значения переменных окружения;
  - версию кода/commit hash, развернутый на сервере.
- Нужно определить частоту backup, срок хранения и процедуру восстановления.

## 2. Что нужно уточнить у ИТ

### Сервер и ОС

- Какая ОС будет использоваться на сервере?
- Есть ли ограничения по установке пакетов?
- Есть ли доступ администратора/root или только ограниченный пользователь?
- Какой объём диска доступен для базы, media-файлов и backup?

### Docker и runtime

- Есть ли Docker на сервере?
- Есть ли Docker Compose?
- Можно ли запускать сервисы в контейнерах?
- Если Docker запрещён, можно ли использовать Python virtualenv и systemd/service unit?

### PostgreSQL

- Есть ли уже установленный PostgreSQL?
- Будет ли база отдельным сервисом или контейнером?
- Кто создаёт пользователя, базу и права доступа?
- Какой формат подключения разрешён: localhost, internal network, отдельный DB-сервер?

### LibreOffice / Р7-Офис / PDF

- Можно ли устанавливать LibreOffice на сервер?
- Можно ли использовать Р7-Офис или другой офисный пакет для конвертации DOCX → PDF?
- Разрешён ли headless-режим конвертации на сервере?
- Нужно ли PDF включать на первом тестовом стенде или оставить выключенным?

### Домен, IP и доступ пользователей

- Какое доменное имя или IP будет использоваться?
- Нужен ли HTTPS на тестовом стенде?
- Кто будет иметь доступ к системе?
- Нужны ли ограничения по сети/VPN/внутреннему контуру?
- Нужна ли интеграция с корпоративной авторизацией на следующих этапах?

### Backup и восстановление

- Кто отвечает за резервное копирование?
- Где будут храниться backup-файлы?
- Как часто выполнять backup?
- Как быстро нужно уметь восстановить систему?
- Нужно ли тестовое восстановление перед запуском пилота?

### Ограничения безопасности

- Можно ли хранить DOCX/PDF на диске сервера?
- Есть ли требования к шифрованию диска или каталога media?
- Есть ли ограничения на исходящие соединения в интернет?
- Нужно ли полностью offline-развёртывание без доступа к package index?
- Какие логи можно хранить и кто имеет к ним доступ?

## 3. Переменные окружения

| Переменная | Назначение | Пример / значение для тестового стенда | Комментарий |
| --- | --- | --- | --- |
| `SECRET_KEY` / `DJANGO_SECRET_KEY` | Секрет Django | `change-me-to-long-random-secret` | В настройках проекта используется `DJANGO_SECRET_KEY`; `SECRET_KEY` можно считать организационным названием секрета. Не хранить реальный секрет в Git. |
| `DEBUG` / `DJANGO_DEBUG` | Режим отладки | `0` | В настройках проекта используется `DJANGO_DEBUG`. На сервере должно быть `0`. |
| `ALLOWED_HOSTS` / `DJANGO_ALLOWED_HOSTS` | Разрешённые домены/IP | `permit-system.example.org,10.0.0.10` | В настройках проекта используется `DJANGO_ALLOWED_HOSTS`. |
| `DATABASE_URL` | Подключение к PostgreSQL | `postgresql://user:password@db:5432/permit_system` | Если не задан, проект использует SQLite; для сервера нужен PostgreSQL. |
| `MEDIA_ROOT` | Каталог media-файлов | `/app/media` | Должен быть постоянным volume/каталогом. |
| `STATIC_ROOT` | Каталог собранной статики | `/app/staticfiles` | Заполняется командой `collectstatic`. |
| `PDF_CONVERTER_ENABLED` | Включение PDF-конвертации | `0` или `1` | По умолчанию выключено. |
| `SOFFICE_PATH` | Путь/имя LibreOffice `soffice` | `soffice` или `/usr/bin/soffice` | Используется только при включённом PDF. |

## 4. Порядок тестового развёртывания

### Вариант A: Docker Compose

1. Клонировать репозиторий:

   ```bash
   git clone <repo-url> auto-eng
   cd auto-eng
   ```

2. Создать `.env` из примера:

   ```bash
   cp .env.example .env
   ```

3. Заполнить `.env` реальными значениями для тестового стенда.

4. Собрать и запустить сервисы:

   ```bash
   docker compose up --build -d
   ```

5. Применить миграции:

   ```bash
   docker compose run --rm web python manage.py migrate
   ```

6. Собрать статические файлы:

   ```bash
   docker compose run --rm web python manage.py collectstatic --noinput
   ```

7. Создать/обновить роли:

   ```bash
   docker compose run --rm web python manage.py setup_roles
   ```

8. Создать demo data для тестовой демонстрации:

   ```bash
   docker compose run --rm web python manage.py seed_demo_data
   ```

9. Создать superuser:

   ```bash
   docker compose run --rm web python manage.py createsuperuser
   ```

10. Проверить, что сервер отвечает по домену/IP из `ALLOWED_HOSTS`.

### Вариант B: Python virtualenv без Docker

1. Клонировать репозиторий:

   ```bash
   git clone <repo-url> auto-eng
   cd auto-eng
   ```

2. Создать `.env` или экспортировать переменные окружения.

3. Создать virtualenv и установить зависимости:

   ```bash
   python -m venv .venv
   source .venv/bin/activate
   python -m pip install -r requirements-dev.txt
   ```

4. Убедиться, что `DATABASE_URL` указывает на PostgreSQL тестового стенда.

5. Выполнить миграции:

   ```bash
   python manage.py migrate
   ```

6. Собрать статику:

   ```bash
   python manage.py collectstatic --noinput
   ```

7. Создать роли:

   ```bash
   python manage.py setup_roles
   ```

8. Создать demo data:

   ```bash
   python manage.py seed_demo_data
   ```

9. Создать superuser:

   ```bash
   python manage.py createsuperuser
   ```

10. Запустить сервер для тестовой проверки:

    ```bash
    python manage.py runserver 0.0.0.0:8000
    ```

11. Для постоянной эксплуатации заменить `runserver` на согласованный production-runner, например gunicorn + systemd/reverse proxy.

## 5. Риски

### Права доступа к серверу

- Может не быть доступа для установки Docker, PostgreSQL, LibreOffice или Python-пакетов.
- Нужно заранее определить ответственного администратора и уровень доступа.
- Для offline-сервера потребуется заранее подготовленный набор пакетов/wheelhouse или готовый Docker image.

### Хранение документов

- DOCX-шаблоны и сгенерированные DOCX/PDF являются значимыми файлами системы.
- Потеря `MEDIA_ROOT` приведёт к потере шаблонов и готовых документов.
- Нужно убедиться, что media хранится на постоянном диске/volume и входит в backup.

### Бэкапы

- Недостаточно делать backup только PostgreSQL: нужно сохранять и media-файлы.
- Нужно проверить восстановление из backup до пилота.
- Нужно хранить информацию о версии кода, соответствующей backup.

### PDF-конвертация

- LibreOffice/`soffice` может быть недоступен из-за ограничений ИТ.
- Даже при установленном LibreOffice возможны отличия форматирования PDF от DOCX.
- PDF лучше включать после отдельной проверки на реальных шаблонах.

### Обновления системы

- Нужно определить процесс обновления: pull/checkout нужного commit, миграции, collectstatic, перезапуск сервиса.
- Перед обновлением нужно делать backup PostgreSQL и media.
- Нужно иметь план отката на предыдущую версию.

### Доступ без интернета

- Если сервер не имеет доступа в интернет, нельзя рассчитывать на `pip install` из package index.
- Нужно подготовить offline wheelhouse или заранее собранный Docker image.
- UI не должен зависеть от внешних CDN; это уже проверено для текущего интерфейса.
- Нужно убедиться, что документация по offline local run и `LOCAL_TEST_CHECKLIST.md` применима к серверному стенду.

## 6. Что считать готовностью к пилотному тесту

Пилотное тестовое развёртывание можно считать готовым, если:

- сервер запускает приложение без `DEBUG`;
- PostgreSQL подключён через `DATABASE_URL`;
- `MEDIA_ROOT` и `STATIC_ROOT` находятся на постоянном хранилище;
- миграции применены;
- статика собрана;
- роли созданы;
- demo data создана только для демонстрации или удалена перед реальными тестами;
- superuser создан и передан ответственному администратору;
- backup PostgreSQL и media настроен;
- доступ пользователей проверен из внутренней сети;
- DOCX-генерация проверена на реальном шаблоне организации;
- PDF-конвертация явно включена и проверена либо осознанно оставлена выключенной.
