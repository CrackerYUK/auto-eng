# AGENTS.md

## Назначение

Этот репозиторий содержит проект `permit-system` — внутреннюю веб-систему для создания, согласования, хранения и печати нарядов-допусков по DOCX-шаблонам.

Файл задаёт правила работы для агентных инструментов и разработчиков. Его область действия — весь репозиторий.

## Текущий стек

- Python и Django для веб-приложения.
- PostgreSQL как основная база данных в Docker Compose и будущих окружениях развёртывания.
- SQLite как локальная база по умолчанию для проверок без Docker, если `POSTGRES_HOST` не задан.
- Docker и Docker Compose для локального запуска полного стека.
- Django templates для веб-интерфейса.
- `docxtpl` / `python-docx` для генерации DOCX-документов по шаблонам.
- `pytest` или стандартные Django tests для тестирования.
- React на текущем этапе не используется и не должен добавляться без отдельного архитектурного решения.

## Структура приложений

- `users` — пользователи, роли и права.
- `permits` — данные и процессы нарядов-допусков.
- `approvals` — согласования и маршруты утверждения.
- `documents` — DOCX-шаблоны, генерация и хранение документов.
- `audit` — журналирование значимых действий.

## Обязательные правила разработки

1. Не менять бизнес-логику без явного запроса.
2. Не дублировать существующие модели, формы, views, urls, templates и сервисы.
3. Перед добавлением новой сущности сначала искать существующую реализацию.
4. Бизнес-логику держать в `services.py`, а не во views.
5. Views должны оркестрировать HTTP-запросы, формы, права доступа и вызовы сервисов, но не содержать доменную логику согласований или генерации документов.
6. Использовать Django templates. Не добавлять React на текущем этапе.
7. Сохранять проект запускаемым через `docker compose up --build`.
8. Не хранить секреты в коде; использовать переменные окружения.
9. При изменениях добавлять или обновлять тесты.
10. Не использовать screenshot-команды в Codex Cloud. Отсутствие браузера, Chromium, Firefox или `wkhtmltoimage` не считается ошибкой проекта.
11. Не считать ошибкой кода ситуацию, когда зависимости не установлены из-за ограничений network/package index.

## Инварианты предметной области

- `DocumentTemplate` — единственный источник правды для DOCX-шаблонов.
- `GeneratedDocument` — единственный источник правды для готовых документов.
- `generate_permit_docx` — основной сервис генерации DOCX.
- Согласования нарядов должны проходить через существующий workflow и сервисы согласования.
- Аудит значимых действий должен сохраняться через существующую модель/механизм `AuditLog`.

## Обязательные проверки перед сдачей изменений

Основные cloud-check команды для Codex:

```bash
python manage.py check
python manage.py test
```

Для локальных проверок без Docker сначала установите dev-зависимости:

```bash
python -m pip install -r requirements-dev.txt
```

Если `manage.py` или зависимости отсутствуют в текущем checkout, либо установка зависимостей невозможна из-за ограничения сети/package index, зафиксировать это в результате как ограничение окружения, а не ошибку кода.

Docker в контейнере Codex может отсутствовать, поэтому `docker compose config` и другие Docker-команды не являются обязательными cloud-check командами.

## Codex Cloud visual limitations

- Do not attempt screenshot generation in Codex Cloud.
- Missing chromium/firefox/wkhtmltoimage is not a project failure.
- UI checks must be done with Django test client:
  - `status_code`;
  - template rendering;
  - expected HTML text;
  - expected buttons/links/forms.
- Manual visual review is performed locally or on the deployment server.

## Рекомендованный порядок работы

1. Изучить текущую структуру проекта.
2. Найти существующие реализации перед добавлением новых файлов или сущностей.
3. Вносить минимальные изменения по задаче.
4. Не изменять модели, миграции и бизнес-логику без отдельного запроса.
5. Запустить обязательные проверки или явно описать, почему они не могут быть выполнены.
6. Обновлять проектную память при изменении архитектуры, workflow или roadmap.
