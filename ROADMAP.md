# ROADMAP.md

## Назначение

Roadmap фиксирует ближайшие направления развития `permit-system` и помогает не терять контекст между итерациями.

## Принципы планирования

- Сначала завершать целостность существующей бизнес-логики, затем добавлять новые функции.
- Не дублировать уже существующие сущности и сервисы.
- Сохранять `DocumentTemplate` единственным источником правды для шаблонов.
- Сохранять `GeneratedDocument` единственным источником правды для готовых документов.
- Развивать UI на Django templates без React.
- Все изменения workflow и документов сопровождать проверками `python manage.py check` и `python manage.py test`.

## Ближайший этап

Следующий этап после подтверждения MVP — стабилизация и улучшение UX без изменения базовой архитектуры. Основные задачи этапа: убрать шероховатости пользовательского сценария, улучшить навигацию и формы, сделать работу с DOCX-шаблонами понятнее, но не добавлять React и не дублировать существующие модели/сервисы.

Ручной выбор конкретного активного `DocumentTemplate` при генерации DOCX входит в ближайшие UX-задачи, но не является blocker текущего MVP: сейчас генерация работает через автоматический выбор последнего активного шаблона с `document_type="permit"`.

## Выполнено

### Demo setup для локальной демонстрации

Реализована management-команда `seed_demo_data`, которая создаёт непроизводственные demo-данные для локального показа системы:

- пользователей `operator`, `master`, `chief`, `admin`;
- группы и роли через существующую команду `setup_roles`;
- demo-участки, оборудование, виды работ, опасности и меры безопасности;
- несколько demo-нарядов в разных статусах.

Demo setup предназначен только для локальной демонстрации и не является механизмом production-наполнения базы.

### Подготовительный PDF-слой

Реализован подготовительный слой для будущего PDF-экспорта:

- `GeneratedDocument.file_pdf` используется как место хранения PDF-файла;
- добавлен сервис `convert_docx_to_pdf`;
- добавлены настройки `PDF_CONVERTER_ENABLED` и `SOFFICE_PATH`;
- PDF-конвертация по умолчанию выключена;
- тесты PDF-слоя используют mock и не требуют установленный LibreOffice/`soffice`.

## Ближайшие задачи

### 1. Стабилизировать MVP и улучшить UX

Цель: пройти основные пользовательские сценарии после MVP и улучшить удобство без изменения архитектуры.

Ожидаемый результат:

- понятные страницы входа, списка, карточки, создания и редактирования наряда;
- более удобные сообщения об ошибках и успешных действиях;
- аккуратная навигация по действиям workflow;
- сохранение текущего подхода Django templates без React.

### 2. Добавить ручной выбор активного DocumentTemplate при генерации DOCX

Цель: дать пользователю возможность выбрать конкретный активный шаблон при генерации DOCX, когда для `document_type="permit"` доступно несколько активных шаблонов.

Ограничения:

- `DocumentTemplate` остаётся единственным источником правды для DOCX-шаблонов;
- генерация продолжает идти через `generate_permit_docx`;
- результат сохраняется через `GeneratedDocument`;
- задача не является blocker текущего MVP, потому что автоматический выбор последнего активного шаблона уже работает.

### 3. Проверить и завершить аудит изменений Permit

Цель: убедиться, что значимые изменения наряда-допуска фиксируются в `AuditLog`.

Ожидаемый результат:

- понятный набор событий аудита для создания, редактирования, согласования и генерации документов;
- отсутствие дублирующих или пропущенных записей;
- тесты на ключевые сценарии аудита.

### 4. Улучшить интерфейс управления DOCX-шаблонами

Цель: сделать управление шаблонами удобным и безопасным для администраторов.

Возможные улучшения:

- список шаблонов с фильтрами/статусами;
- просмотр метаданных шаблона;
- понятная загрузка новой версии или нового шаблона;
- валидация типа файла;
- отображение признака активного шаблона.

### 5. Добавить версионирование шаблонов

Цель: обеспечить историю изменений DOCX-шаблонов и воспроизводимость уже созданных документов.

Архитектурные ограничения:

- `DocumentTemplate` остаётся источником правды для шаблонов;
- нужно заранее определить модель версионирования: поля версии в `DocumentTemplate` или отдельная связанная сущность;
- `GeneratedDocument` должен сохранять связь с конкретной версией/экземпляром шаблона, использованным при генерации.

### 6. Добавить фильтры и поиск нарядов

Цель: упростить работу с большим количеством нарядов.

Базовые фильтры:

- статус;
- дата создания/обновления;
- роль/ответственный участник;
- зона работ;
- оборудование;
- вид работ;
- текстовый поиск по номеру/описанию.

### 7. Добавить пользовательский PDF-экспорт

Цель: обеспечить получение наряда в PDF-формате из пользовательского интерфейса или отдельного workflow поверх уже подготовленного PDF-слоя.

Уже подготовлено:

- поле `GeneratedDocument.file_pdf`;
- настройки `PDF_CONVERTER_ENABLED` и `SOFFICE_PATH`;
- сервисный интерфейс `convert_docx_to_pdf`;
- безопасные ошибки, если конвертер выключен или LibreOffice/`soffice` отсутствует.

Оставшиеся решения:

- где запускать конвертацию в UI/API;
- нужен ли отдельный permission/role для PDF-экспорта;
- делать ли конвертацию синхронной или отложенной;
- как отображать готовый PDF рядом с DOCX.

Важно: отсутствие браузера, Chromium, Firefox или `wkhtmltoimage` не является ошибкой проекта.

### 8. Подготовить развёртывание на сервере организации

Цель: подготовить проект к эксплуатации во внутренней инфраструктуре.

Направления работ:

- production-настройки Django;
- переменные окружения;
- PostgreSQL backup/restore;
- хранение media-файлов;
- сбор статических файлов;
- Docker Compose для сервера;
- инструкция установки и обновления;
- базовый мониторинг и журналирование.

## Отложенные решения

- Нужно ли добавлять отдельную модель версии шаблона или достаточно расширить `DocumentTemplate`.
- Какой инструмент использовать для DOCX → PDF в серверной среде организации.
- Какие дополнительные роли потребуются после пилотной эксплуатации.
- Нужны ли уведомления о согласованиях и через какой канал.
