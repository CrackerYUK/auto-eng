# ROADMAP.md

## Назначение

Roadmap фиксирует ближайшие направления развития `permit-system` и помогает не терять контекст между итерациями.

## Принципы планирования

- Сначала завершать целостность существующей бизнес-логики, затем добавлять новые функции.
- Не дублировать уже существующие сущности и сервисы.
- Сохранять `DocumentTemplate` единственным источником правды для шаблонов.
- Сохранять `GeneratedDocument` единственным источником правды для готовых документов.
- Развивать UI на Django templates без React.
- Все изменения workflow и документов сопровождать проверками `python manage.py check` и `python manage.py test`.

## Ближайшие задачи

### 1. Проверить и завершить аудит изменений Permit

Цель: убедиться, что значимые изменения наряда-допуска фиксируются в `AuditLog`.

Ожидаемый результат:

- понятный набор событий аудита для создания, редактирования, согласования и генерации документов;
- отсутствие дублирующих или пропущенных записей;
- тесты на ключевые сценарии аудита.

### 2. Улучшить интерфейс управления DOCX-шаблонами

Цель: сделать управление шаблонами удобным и безопасным для администраторов.

Возможные улучшения:

- список шаблонов с фильтрами/статусами;
- просмотр метаданных шаблона;
- понятная загрузка новой версии или нового шаблона;
- валидация типа файла;
- отображение признака активного шаблона.

### 3. Добавить версионирование шаблонов

Цель: обеспечить историю изменений DOCX-шаблонов и воспроизводимость уже созданных документов.

Архитектурные ограничения:

- `DocumentTemplate` остаётся источником правды для шаблонов;
- нужно заранее определить модель версионирования: поля версии в `DocumentTemplate` или отдельная связанная сущность;
- `GeneratedDocument` должен сохранять связь с конкретной версией/экземпляром шаблона, использованным при генерации.

### 4. Добавить фильтры и поиск нарядов

Цель: упростить работу с большим количеством нарядов.

Базовые фильтры:

- статус;
- дата создания/обновления;
- роль/ответственный участник;
- зона работ;
- оборудование;
- вид работ;
- текстовый поиск по номеру/описанию.

### 5. Добавить PDF-экспорт

Цель: обеспечить получение наряда в PDF-формате.

Варианты реализации нужно выбирать отдельно после анализа окружения:

- конвертация DOCX в PDF на сервере;
- интеграция с установленным офисным пакетом/конвертером;
- отложенная генерация PDF;
- хранение результата через `GeneratedDocument` или расширение существующей модели готовых документов.

Важно: отсутствие браузера, Chromium, Firefox или `wkhtmltoimage` не является ошибкой проекта.

### 6. Подготовить развёртывание на сервере организации

Цель: подготовить проект к эксплуатации во внутренней инфраструктуре.

Направления работ:

- production-настройки Django;
- переменные окружения;
- PostgreSQL backup/restore;
- хранение media-файлов;
- сбор статических файлов;
- Docker Compose для сервера;
- инструкция установки и обновления;
- базовый мониторинг и журналирование.

## Отложенные решения

- Нужно ли добавлять отдельную модель версии шаблона или достаточно расширить `DocumentTemplate`.
- Какой инструмент использовать для DOCX → PDF в серверной среде организации.
- Какие дополнительные роли потребуются после пилотной эксплуатации.
- Нужны ли уведомления о согласованиях и через какой канал.
