# ARCHITECTURE.md

## Назначение документа

Документ описывает архитектурные правила `permit-system` и фиксирует решения, которые нужно учитывать при разработке новых функций.

## Высокоуровневая архитектура

`permit-system` строится как классическое Django-приложение:

- Django обрабатывает HTTP-запросы, маршрутизацию, формы, templates и права доступа.
- PostgreSQL хранит данные предметной области, пользователей, workflow, аудит, шаблоны и готовые документы.
- Docker/Docker Compose используются для локального и серверного запуска.
- DOCX-документы генерируются через `docxtpl` / `python-docx`.
- Интерфейс строится на Django templates без React.

## Слои приложения

### Models

Models описывают структуру данных и связи между сущностями. Запрещено создавать дубликаты существующих моделей.

Ключевые модели проекта:

- `Permit`;
- `ApprovalAction`;
- `AuditLog`;
- `DocumentTemplate`;
- `GeneratedDocument`;
- `WorkArea`;
- `Equipment`;
- `WorkType`;
- `Hazard`;
- `SafetyMeasure`.

### Services

Services содержат бизнес-логику:

- переходы workflow;
- проверки допустимости действий;
- согласования;
- генерацию документов;
- аудит доменных операций.

Правило: бизнес-логику держать в `services.py`, а не во views.

Основной сервис генерации DOCX:

- `generate_permit_docx`.

### Views

Views отвечают за:

- приём HTTP-запросов;
- проверку авторизации и прав доступа;
- обработку форм;
- вызов сервисов;
- выбор template/redirect/response.

Views не должны содержать самостоятельную реализацию workflow согласований или генерации документов.

### Templates

UI строится на Django templates. Новые страницы должны переиспользовать существующие базовые шаблоны, partials и стили, если они есть.

React не добавляется на текущем этапе.

### Documents

`DocumentTemplate` — единственный источник правды для DOCX-шаблонов. Не нужно создавать альтернативные модели, таблицы или файловые реестры шаблонов без отдельного архитектурного решения.

`GeneratedDocument` — единственный источник правды для готовых документов. Любая выдача готового файла должна опираться на эту сущность или существующий сервисный слой.

## Workflow согласований

Согласование нарядов должно выполняться через существующий workflow и сервисы согласования.

При добавлении новых действий нужно проверить:

1. существует ли уже нужное действие;
2. какая роль имеет право его выполнять;
3. какой статус наряда является исходным;
4. какой статус должен быть после действия;
5. требуется ли запись в `ApprovalAction`;
6. требуется ли запись в `AuditLog`;
7. нужно ли перегенерировать документ.

## Аудит

Значимые изменения `Permit` и действия согласования должны быть отражены в аудите через существующий механизм `AuditLog`.

Отдельная ближайшая задача — проверить и завершить аудит изменений `Permit`.

## Шаблоны документов

Правила:

- не создавать второй источник правды для шаблонов;
- хранить и выбирать шаблоны через `DocumentTemplate`;
- генерировать документы через `generate_permit_docx`;
- результаты генерации фиксировать через `GeneratedDocument`;
- при будущем версионировании шаблонов сохранить совместимость с уже сгенерированными документами.

## Ограничения текущего этапа

- Не добавлять React.
- Не переносить UI на SPA.
- Не дублировать модели, формы, views, urls, templates и сервисы.
- Не менять бизнес-логику без явного запроса.
- Не вводить альтернативный механизм генерации документов в обход `generate_permit_docx`.

## Проверки

Для полноценного Django checkout обязательны:

```bash
python manage.py check
python manage.py test
```

Отсутствие браузера, Chromium, Firefox или `wkhtmltoimage` не считается ошибкой проекта. Screenshot-команды в Codex Cloud не являются обязательными и не должны использоваться как блокирующая проверка.
