# LOCAL_TEST_CHECKLIST.md

## Назначение

Чек-лист предназначен для проверки `permit-system` на ноутбуке без доступа к интернету. Он помогает последовательно проверить offline local run, Django-команды, demo setup, MVP-цикл наряда-допуска, DOCX-генерацию и безопасное поведение PDF-слоя без LibreOffice.

> Важно: не используйте production-данные для локальной проверки. Demo-пользователи и demo-наряды предназначены только для демонстрации.

## 1. Проверка окружения

### Python

- [ ] Проверить версию Python:

  ```bash
  python --version
  ```

- [ ] Убедиться, что версия Python соответствует локальному окружению проекта.

### Наличие проекта

- [ ] Перейти в корень репозитория:

  ```bash
  cd /path/to/auto-eng
  ```

- [ ] Проверить наличие ключевых файлов:

  ```bash
  test -f manage.py
  test -f requirements-dev.txt
  test -f README.md
  test -d permit_system
  test -d permits
  test -d documents
  ```

### Наличие wheelhouse для offline-установки

- [ ] Проверить, что рядом с проектом или внутри проекта есть каталог с заранее скачанными wheel-пакетами, например `wheelhouse/`:

  ```bash
  test -d wheelhouse
  ```

- [ ] Если `wheelhouse/` отсутствует, зафиксировать это как проблему подготовки offline-окружения, а не как ошибку Django-кода.

### Создание virtualenv

- [ ] Создать виртуальное окружение:

  ```bash
  python -m venv .venv
  ```

- [ ] Активировать виртуальное окружение:

  ```bash
  source .venv/bin/activate
  ```

- [ ] Проверить, что используется Python из `.venv`:

  ```bash
  which python
  python --version
  ```

### Offline-установка зависимостей

- [ ] Установить зависимости из локального `wheelhouse/` без обращения к интернету:

  ```bash
  python -m pip install --no-index --find-links=wheelhouse -r requirements-dev.txt
  ```

- [ ] Если установка падает из-за отсутствующего wheel-файла, записать имя пакета и подготовить wheelhouse заново в окружении с интернетом.

## 2. Проверка Django

> Для проверки без PostgreSQL убедитесь, что `DATABASE_URL` не задан. Тогда проект использует SQLite `db.sqlite3`.

- [ ] Отключить `DATABASE_URL` для локального SQLite-запуска:

  ```bash
  unset DATABASE_URL
  ```

- [ ] Проверить конфигурацию Django:

  ```bash
  python manage.py check
  ```

- [ ] Запустить тесты:

  ```bash
  python manage.py test
  ```

- [ ] Применить миграции:

  ```bash
  python manage.py migrate
  ```

## 4. Проверка запуска

- [ ] Запустить локальный Django-сервер:

  ```bash
  python manage.py runserver
  ```

- [ ] Открыть приложение в браузере:

  ```text
  http://127.0.0.1:8000
  ```

- [ ] Проверить, что страница открывается без внешнего интернета и без ошибок загрузки CDN/Google Fonts/внешних UI-ресурсов.

## 5. Проверка MVP

### Operator

- [ ] Войти как `operator`.
- [ ] Открыть список нарядов.
- [ ] Создать новый `Permit`.
- [ ] Убедиться, что новый `Permit` создан в статусе `draft`.
- [ ] Отредактировать `Permit` в статусе `draft`.
- [ ] Отправить `Permit` на согласование.
- [ ] Убедиться, что статус изменился на `submitted`.

### Master

- [ ] Выйти из `operator`.
- [ ] Войти как `master`.
- [ ] Найти наряд в статусе `submitted`.
- [ ] Выполнить согласование мастером или возврат на доработку.
- [ ] Если выполнено согласование, убедиться, что статус изменился на `approved_by_master`.
- [ ] Если выполнен возврат, убедиться, что статус изменился на `returned`.

### Chief

- [ ] Выйти из `master`.
- [ ] Войти как `chief`.
- [ ] Найти наряд в статусе `approved_by_master`.
- [ ] Утвердить наряд.
- [ ] Убедиться, что статус изменился на `approved_by_chief`.

### DOCX-шаблон и генерация документа

- [ ] Проверить наличие активного `DocumentTemplate` для `document_type="permit"`.
- [ ] Если шаблона нет, загрузить тестовый DOCX-шаблон через Django admin.
- [ ] Открыть карточку наряда.
- [ ] Выбрать `DocumentTemplate`, если в интерфейсе доступен ручной выбор.
- [ ] Сгенерировать DOCX.
- [ ] Убедиться, что появился `GeneratedDocument`.
- [ ] Скачать DOCX.
- [ ] Открыть DOCX локально и проверить заполнение ключевых полей.

## 6. Проверка PDF

- [ ] Убедиться, что PDF-конвертация по умолчанию выключена:

  ```bash
  python manage.py shell -c "from django.conf import settings; print(settings.PDF_CONVERTER_ENABLED)"
  ```

- [ ] Ожидаемое значение по умолчанию:

  ```text
  False
  ```

- [ ] Убедиться, что отсутствие LibreOffice/`soffice` не ломает DOCX-генерацию.
- [ ] Сгенерировать DOCX при выключенном `PDF_CONVERTER_ENABLED`.
- [ ] Проверить, что DOCX скачивается успешно.
- [ ] Если нажимается PDF-действие при выключенной конвертации, убедиться, что пользователь видит понятное сообщение, а не ошибку 500.

## 7. Что записывать при ошибках

При любой ошибке фиксировать максимально полный контекст:

- [ ] Команда, которая выполнялась.
- [ ] Полный traceback или полный текст ошибки.
- [ ] Файл, URL или страница, где ошибка проявилась.
- [ ] Какой пользователь был активен:
  - `operator`;
  - `master`;
  - `chief`;
  - `admin`;
  - anonymous/неавторизованный пользователь.
- [ ] Какой `Permit` проверялся:
  - id;
  - номер;
  - текущий статус.
- [ ] Какое действие выполнялось:
  - создание;
  - редактирование;
  - отправка на согласование;
  - согласование мастером;
  - возврат;
  - утверждение начальником;
  - генерация DOCX;
  - скачивание DOCX;
  - генерация/скачивание PDF.
- [ ] Был ли задан `DATABASE_URL`.
- [ ] Использовался ли SQLite или PostgreSQL.
- [ ] Был ли доступ к интернету отключён полностью.
- [ ] Был ли установлен LibreOffice/`soffice`.

## Критерий успешной локальной проверки

Проверка считается успешной, если:

- [ ] зависимости установлены offline из `wheelhouse/`;
- [ ] `python manage.py check` проходит;
- [ ] `python manage.py test` проходит;
- [ ] миграции применяются;
- [ ] demo data создаётся;
- [ ] приложение открывается по `http://127.0.0.1:8000` без внешнего интернета;
- [ ] MVP-цикл operator → master → chief проходит;
- [ ] DOCX генерируется и скачивается;
- [ ] выключенный PDF-слой не ломает DOCX-сценарий.
