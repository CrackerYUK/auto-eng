# PROJECT_STATE.md

## Проект

`permit-system` — внутренняя веб-система для создания, согласования, хранения и печати нарядов-допусков по DOCX-шаблонам.

## Фактическое состояние после merge

Django-проект находится в корне репозитория. Команды Django нужно запускать из корня, где расположен `manage.py`.

Ключевые файлы и директории текущего checkout:

- `manage.py` — точка входа Django management-команд.
- `permit_system/` — настройки проекта, корневые URL, ASGI/WSGI.
- `requirements.txt` — runtime-зависимости.
- `requirements-dev.txt` — зависимости для разработки и тестов.
- `Dockerfile` — образ Django-приложения.
- `docker-compose.yml` — локальный стек Django + PostgreSQL.
- `templates/` — Django templates базового веб-интерфейса.
- `tests/` — проектные тесты.
- `users/management/commands/seed_demo_data.py` — demo setup для локальной демонстрации.

## Актуальные Django-приложения

- `users` — пользователи, роли, базовые группы и management-команда `setup_roles`.
- `permits` — наряды-допуски, справочники, формы, views и templates для базового веб-интерфейса.
- `approvals` — модели и сервисы workflow согласования.
- `documents` — DOCX-шаблоны, сгенерированные документы и сервис генерации.
- `audit` — журналирование значимых действий пользователей.

## Статус MVP

MVP-цикл наряда-допуска реализован и покрыт автоматическими тестами: пользователь может войти в систему, оператор может создать черновик наряда, отредактировать его в разрешённых статусах, отправить на проверку, мастер и начальник могут выполнять согласования/возвраты/отклонения, действия сохраняются в `ApprovalAction` и `AuditLog`, DOCX генерируется и скачивается через веб-интерфейс.

Единственное известное ограничение MVP: в интерфейсе генерации DOCX пока нет ручного выбора конкретного активного `DocumentTemplate`. Сейчас используется автоматический выбор последнего активного шаблона с `document_type="permit"`. Ручной выбор шаблона зафиксирован как ближайшее UX-улучшение, но не является blocker для текущего MVP.

## Реализованные функции

### Наряды-допуски

- Реализована модель `Permit`.
- Реализованы статусы жизненного цикла `PermitStatus`.
- Реализованы справочники:
  - `WorkArea`;
  - `Equipment`;
  - `WorkType`;
  - `Hazard`;
  - `SafetyMeasure`.
- Реализована форма `PermitForm`.
- Добавлен ручной ввод нестандартных данных для `Permit`: ответственный руководитель работ, производитель работ, характер работ, дополнительные условия и дополнительные меры безопасности. Существующее поле `work_location` используется как ручное место проведения работ.
- Добавлен справочник персонала: `PersonnelGroup` и `Personnel`. Записи персонала не являются пользователями сайта и не получают логин/пароль.
- Добавлена гибкая модель участников наряда `PermitParticipant`: один Permit может иметь несколько ответственных руководителей, производителей работ, исполнителей, членов бригады, допускающих, наблюдающих и прочих участников.
- На страницах создания и редактирования Permit есть блок **«Участники и ответственные»** с выбором работника из справочника, ручным вводом участника, ролью и примечанием.
- На странице просмотра Permit участники показываются сгруппированно по ролям.
- В формах участников по умолчанию доступны только активные работники; уже выбранный неактивный работник остаётся отображаемым для старых нарядов. Есть лёгкий endpoint `/personnel/search/?q=`, но полноценный autocomplete в форме остаётся pending UX improvement.
- Реализован базовый веб-интерфейс на Django templates:
  - список нарядов;
  - карточка наряда;
  - создание наряда;
  - редактирование наряда;
  - выполнение доступных действий workflow из интерфейса.

### Согласования

- Реализована модель `ApprovalAction`.
- Реализованы сервисы согласования и переходов workflow.
- Поддерживаются базовые роли:
  - `operator`;
  - `master`;
  - `chief`;
  - `admin`.

### Аудит

- Реализована модель `AuditLog`.
- Реализуется аудит значимых действий с нарядами и workflow.

### Demo setup

- Реализована management-команда `seed_demo_data` для локальной демонстрации системы.
- Команда создаёт только непроизводственные demo-данные:
  - тестовых пользователей `operator`, `master`, `chief`, `admin`;
  - группы и роли через существующий механизм `setup_roles`;
  - demo-участки `WorkArea`;
  - demo-оборудование `Equipment`;
  - demo-виды работ `WorkType`;
  - demo-опасности `Hazard`;
  - demo-меры безопасности `SafetyMeasure`;
  - группы персонала `PersonnelGroup`;
  - demo-работников `Personnel`;
  - несколько demo-нарядов `Permit` в разных статусах;
  - участников demo-нарядов `PermitParticipant`;
  - demo DOCX-шаблон с транслит-переменными.
- Demo setup предназначен только для локальной демонстрации и не должен использоваться как источник production-данных.

### Offline local run и UI-зависимости

- Offline local run подготовлен: если `DATABASE_URL` не задан, проект использует локальный SQLite-файл `db.sqlite3`.
- Для PostgreSQL нужно явно задать `DATABASE_URL`; Docker/test deployment продолжает использовать PostgreSQL через переменные окружения.
- Проверено, что интерфейс не зависит от внешних CDN, Google Fonts, `gstatic`, `unpkg`, Bootstrap CDN, Font Awesome, Cloudflare и других внешних UI-ресурсов.
- После установки Python-зависимостей интерфейс должен открываться локально без доступа к интернету, потому что страницы используют Django templates и локальные стили проекта.

### DOCX-документы

- Реализована модель `DocumentTemplate` — источник DOCX-шаблонов.
- Реализована модель `GeneratedDocument` — источник готовых документов.
- Реализован основной сервис генерации `generate_permit_docx`.
- Основная система DOCX-переменных переведена на безопасные транслит-переменные (`nomer_naryada`, `mesto_rabot`, `uchastniki_rabot` и др.).
- Старые английские переменные в контексте сохранены для совместимости, а кириллические имена переменных больше не считаются рекомендуемыми для новых шаблонов.
- В DOCX context добавлены переменные участников наряда: ответственные руководители, производители работ, исполнители, члены бригады, допускающие, наблюдающие, прочие и общий список участников.
- Реализована генерация DOCX из карточки наряда при разрешённом статусе и роли пользователя.
- При генерации автоматически выбирается последний активный `DocumentTemplate` с `document_type="permit"`.
- Реализовано скачивание сгенерированного DOCX через веб-интерфейс.
- Управление `DocumentTemplate` и `GeneratedDocument` доступно через Django admin; для `DocumentTemplate` добавлена проверка шаблона на demo-данных без создания `GeneratedDocument`. Ручной выбор конкретного активного шаблона в пользовательском интерфейсе генерации остаётся known limitation / next improvement.

### PDF-экспорт

- Подготовительный PDF-слой реализован.
- `GeneratedDocument.file_pdf` уже используется как место хранения будущего PDF-файла.
- Добавлен сервис `convert_docx_to_pdf` для конвертации DOCX в PDF через LibreOffice/`soffice`.
- `PDF_CONVERTER_ENABLED` по умолчанию выключен, поэтому PDF-конвертация не запускается без явного включения.
- `SOFFICE_PATH` задаёт путь или имя исполняемого файла LibreOffice/`soffice` на сервере.
- `generate_permit_docx` не изменён и продолжает отвечать за основную DOCX-генерацию.
- PDF-конвертация покрыта тестами через mock; реальный LibreOffice в тестовой среде не требуется.

## Стабильные архитектурные решения

- Основной backend-фреймворк: Django.
- Основная база данных: PostgreSQL в Docker Compose и будущих окружениях развёртывания.
- Локальная база без Docker: SQLite, если `DATABASE_URL` не задан.
- Среда запуска: Docker/Docker Compose.
- UI: Django templates.
- Генерация документов: `docxtpl` / `python-docx`.
- Текущий frontend-подход: без React.

## Инварианты проекта

- Не дублировать существующие модели, формы, views, urls, templates и сервисы.
- Перед добавлением новой сущности сначала искать существующую реализацию.
- `DocumentTemplate` — единственный источник правды для DOCX-шаблонов.
- `GeneratedDocument` — единственный источник правды для готовых документов.
- `generate_permit_docx` — основной сервис генерации DOCX.
- Бизнес-логику держать в `services.py`, а не во views.
- Не добавлять React на текущем этапе.
- Не использовать screenshot-команды в Codex Cloud.

## Обязательные проверки

Перед сдачей изменений нужно выполнить из корня репозитория:

```bash
python manage.py check
python manage.py test
```

Если зависимости не установлены, сначала выполнить:

```bash
python -m pip install -r requirements-dev.txt
```

Если зависимости не устанавливаются из-за ограничений network/package index, это ограничение окружения, а не ошибка кода.
