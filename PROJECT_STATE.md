# PROJECT_STATE.md

## Проект

`permit-system` — внутренняя веб-система для создания, согласования, хранения и печати нарядов-допусков по DOCX-шаблонам.

## Фактическое состояние после merge

Django-проект находится в корне репозитория. Команды Django нужно запускать из корня, где расположен `manage.py`.

Ключевые файлы и директории текущего checkout:

- `manage.py` — точка входа Django management-команд.
- `permit_system/` — настройки проекта, корневые URL, ASGI/WSGI.
- `requirements.txt` — runtime-зависимости.
- `requirements-dev.txt` — зависимости для разработки и тестов.
- `Dockerfile` — образ Django-приложения.
- `docker-compose.yml` — локальный стек Django + PostgreSQL.
- `templates/` — Django templates базового веб-интерфейса.
- `tests/` — проектные тесты.

## Актуальные Django-приложения

- `users` — пользователи, роли, базовые группы и management-команда `setup_roles`.
- `permits` — наряды-допуски, справочники, формы, views и templates для базового веб-интерфейса.
- `approvals` — модели и сервисы workflow согласования.
- `documents` — DOCX-шаблоны, сгенерированные документы и сервис генерации.
- `audit` — журналирование значимых действий пользователей.

## Статус MVP

MVP-цикл наряда-допуска реализован и покрыт автоматическими тестами: пользователь может войти в систему, оператор может создать черновик наряда, отредактировать его в разрешённых статусах, отправить на проверку, мастер и начальник могут выполнять согласования/возвраты/отклонения, действия сохраняются в `ApprovalAction` и `AuditLog`, DOCX генерируется и скачивается через веб-интерфейс.

Единственное известное ограничение MVP: в интерфейсе генерации DOCX пока нет ручного выбора конкретного активного `DocumentTemplate`. Сейчас используется автоматический выбор последнего активного шаблона с `document_type="permit"`. Ручной выбор шаблона зафиксирован как ближайшее UX-улучшение, но не является blocker для текущего MVP.

## Реализованные функции

### Наряды-допуски

- Реализована модель `Permit`.
- Реализованы статусы жизненного цикла `PermitStatus`.
- Реализованы справочники:
  - `WorkArea`;
  - `Equipment`;
  - `WorkType`;
  - `Hazard`;
  - `SafetyMeasure`.
- Реализована форма `PermitForm`.
- Реализован базовый веб-интерфейс на Django templates:
  - список нарядов;
  - карточка наряда;
  - создание наряда;
  - редактирование наряда;
  - выполнение доступных действий workflow из интерфейса.

### Согласования

- Реализована модель `ApprovalAction`.
- Реализованы сервисы согласования и переходов workflow.
- Поддерживаются базовые роли:
  - `operator`;
  - `master`;
  - `chief`;
  - `admin`.

### Аудит

- Реализована модель `AuditLog`.
- Реализуется аудит значимых действий с нарядами и workflow.

### DOCX-документы

- Реализована модель `DocumentTemplate` — источник DOCX-шаблонов.
- Реализована модель `GeneratedDocument` — источник готовых документов.
- Реализован основной сервис генерации `generate_permit_docx`.
- Реализована генерация DOCX из карточки наряда при разрешённом статусе и роли пользователя.
- При генерации автоматически выбирается последний активный `DocumentTemplate` с `document_type="permit"`.
- Реализовано скачивание сгенерированного DOCX через веб-интерфейс.
- Управление `DocumentTemplate` и `GeneratedDocument` доступно через Django admin; ручной выбор конкретного активного шаблона в пользовательском интерфейсе генерации остаётся known limitation / next improvement.

## Стабильные архитектурные решения

- Основной backend-фреймворк: Django.
- Основная база данных: PostgreSQL в Docker Compose и будущих окружениях развёртывания.
- Локальная база без Docker: SQLite, если `POSTGRES_HOST` не задан.
- Среда запуска: Docker/Docker Compose.
- UI: Django templates.
- Генерация документов: `docxtpl` / `python-docx`.
- Текущий frontend-подход: без React.

## Инварианты проекта

- Не дублировать существующие модели, формы, views, urls, templates и сервисы.
- Перед добавлением новой сущности сначала искать существующую реализацию.
- `DocumentTemplate` — единственный источник правды для DOCX-шаблонов.
- `GeneratedDocument` — единственный источник правды для готовых документов.
- `generate_permit_docx` — основной сервис генерации DOCX.
- Бизнес-логику держать в `services.py`, а не во views.
- Не добавлять React на текущем этапе.
- Не использовать screenshot-команды в Codex Cloud.

## Обязательные проверки

Перед сдачей изменений нужно выполнить из корня репозитория:

```bash
python manage.py check
python manage.py test
```

Если зависимости не установлены, сначала выполнить:

```bash
python -m pip install -r requirements-dev.txt
```

Если зависимости не устанавливаются из-за ограничений network/package index, это ограничение окружения, а не ошибка кода.
