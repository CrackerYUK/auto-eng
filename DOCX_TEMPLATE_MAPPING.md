# DOCX_TEMPLATE_MAPPING.md

## Назначение

Документ фиксирует начальную карту соответствия между переменными DOCX-шаблона и данными системы `permit-system` для наряда-допуска `Permit`.

Карта нужна для проверки реального DOCX-шаблона организации перед загрузкой в `DocumentTemplate` и генерацией через основной сервис `generate_permit_docx`.

> Важно: этот файл описывает соглашение по шаблонам и проверочный список. Он не меняет модели, бизнес-логику или сервис генерации.

## Общие правила для DOCX-шаблона

- Переменные в DOCX должны соответствовать контексту, который передаётся в `docxtpl`.
- Основной объект шаблона — `permit`.
- Для справочников и пользователей безопаснее использовать текстовые поля `*_name`, если они доступны в контексте, чтобы в документ попадали человекочитаемые значения.
- Для списков опасностей и мер безопасности нужно заранее выбрать формат: строка через запятую, список строк или таблица.
- Даты в DOCX-шаблоне нужно проверить отдельно: может потребоваться форматирование средствами `docxtpl`/Jinja или предварительная подготовка отображаемых значений в сервисе.

## Карта переменных DOCX-шаблона

| Переменная шаблона | Источник данных в системе | Модель/поле | Обязательное поле | Пример значения | Комментарий |
| --- | --- | --- | --- | --- | --- |
| `permit.number` | Номер наряда-допуска | `Permit.number` | да | `ND-2026-001` | Уникальный номер Permit. |
| `permit.status` | Текущий статус lifecycle | `Permit.status` | да | `draft` | Машинное значение статуса. Для человекочитаемого вывода лучше проверить доступность `permit.status_display`. |
| `permit.work_area` | Участок работ | `Permit.work_area` → `WorkArea.__str__()` / `WorkArea.name` | да | `Цех 1` | Для стабильного текста предпочтительно использовать имя участка. |
| `permit.equipment` | Оборудование | `Permit.equipment` → `Equipment.__str__()` / `Equipment.name` | нет | `Насос H-101 (H-101)` | Оборудование может быть пустым. В шаблоне нужно проверить отображение пустого значения. |
| `permit.work_type` | Вид работ | `Permit.work_type` → `WorkType.__str__()` / `WorkType.name` | да | `Огневые работы` | Для стабильного текста предпочтительно использовать имя вида работ. |
| `permit.start_date` | Дата/время начала работ | `Permit.work_starts_at` | да | `13.06.2026 09:00` | В текущей модели поле называется `work_starts_at`; переменную `permit.start_date` нужно сверить с фактическим контекстом шаблона перед использованием. |
| `permit.end_date` | Дата/время окончания работ | `Permit.work_ends_at` | да | `13.06.2026 18:00` | В текущей модели поле называется `work_ends_at`; переменную `permit.end_date` нужно сверить с фактическим контекстом шаблона перед использованием. |
| `permit.work_location` | Место выполнения работ | `Permit.work_location` | да | `Отм. +6.000, площадка обслуживания` | Короткое текстовое поле. Проверить переносы/длину строки в шаблоне. |
| `permit.work_description` | Описание работ | `Permit.work_description` | да | `Ремонт запорной арматуры на линии подачи` | Длинный текст. Нужно проверить перенос строк и сохранение форматирования. |
| `permit.responsible_manager` | Ответственный руководитель | `Permit.responsible_manager` → user `__str__()` / username | да | `chief` | Для человекочитаемого вывода может потребоваться ФИО, если оно будет заполнено в пользователе. |
| `permit.work_producer` | Производитель работ / непосредственный руководитель | `Permit.work_supervisor` | да | `master` | В текущей модели поле называется `work_supervisor`; название `work_producer` нужно согласовать с реальным шаблоном или использовать фактическую переменную контекста. |
| `permit.hazards` | Опасности | `Permit.hazards` → `ManyToManyField(Hazard)` | нет | `Высота, Электрическое напряжение` | Для DOCX обычно удобнее строка имён или цикл по списку. Нужно проверить формат в реальном шаблоне. |
| `permit.safety_measures` | Меры безопасности | `Permit.safety_measures` → `ManyToManyField(SafetyMeasure)` | нет | `Отключить питание, Оформить ограждение` | Для DOCX обычно удобнее строка имён или цикл по списку. Нужно проверить формат в реальном шаблоне. |
| `permit.created_by` | Автор наряда | `Permit.created_by` → user `__str__()` / username | да | `operator` | Обычно demo/login-значение; для production может потребоваться ФИО. |
| `permit.created_at` | Дата/время создания наряда | `Permit.created_at` | да | `13.06.2026 08:30` | Значение создаётся автоматически. Проверить формат даты/времени в DOCX. |

## Фактические полезные переменные текущего контекста

Перед утверждением реального шаблона нужно сверить переменные выше с фактическим контекстом генерации. На текущем этапе сервис генерации подготавливает также следующие полезные значения:

| Переменная | Назначение |
| --- | --- |
| `permit.id` | ID наряда в базе. |
| `permit.status_display` | Человекочитаемое название статуса Django choices. |
| `permit.work_starts_at` | Фактическое поле начала работ. |
| `permit.work_ends_at` | Фактическое поле окончания работ. |
| `permit.work_area_name` | Название участка. |
| `permit.equipment_name` | Название оборудования или пустая строка. |
| `permit.equipment_code` | Код оборудования или пустая строка. |
| `permit.work_type_name` | Название вида работ. |
| `permit.hazard_names` | Список названий опасностей. |
| `permit.hazard_names_text` | Названия опасностей одной строкой через запятую. |
| `permit.safety_measure_names` | Список названий мер безопасности. |
| `permit.safety_measure_names_text` | Названия мер безопасности одной строкой через запятую. |
| `permit.responsible_manager_name` | Username ответственного руководителя. |
| `permit.work_supervisor` | Пользователь-производитель/руководитель работ в текущей модели. |
| `permit.work_supervisor_name` | Username производителя/руководителя работ. |
| `permit.created_by_name` | Username автора наряда. |
| `permit.updated_at` | Дата/время последнего обновления наряда. |

## Что нужно проверить на реальном шаблоне

Перед использованием реального DOCX-шаблона организации нужно проверить:

- [ ] Названия переменных совпадают с фактическим контекстом `generate_permit_docx`.
- [ ] В шаблоне нет опечаток в именах переменных.
- [ ] Таблицы DOCX корректно сохраняют структуру после генерации.
- [ ] Перенос строк в длинных текстах выглядит читаемо.
- [ ] Поля подписей расположены корректно и не съезжают после генерации.
- [ ] Даты отображаются в нужном формате организации.
- [ ] Списки опасностей отображаются в согласованном формате: строка, список или таблица.
- [ ] Списки мер безопасности отображаются в согласованном формате: строка, список или таблица.
- [ ] Длинные тексты в `work_description` не ломают таблицы и страницы.
- [ ] Форматирование после генерации соответствует исходному шаблону: шрифты, интервалы, границы таблиц, выравнивание.
- [ ] Пустое оборудование отображается корректно, если наряд создаётся без `Equipment`.
- [ ] ФИО/username пользователей отображаются в ожидаемом виде.
- [ ] Готовый DOCX открывается в Microsoft Word/LibreOffice без предупреждений о повреждении файла.

## Рекомендации по первому реальному шаблону

1. Начать с минимального шаблона, где есть только основные поля: номер, статус, участок, оборудование, вид работ, даты, место и описание работ.
2. Проверить генерацию и скачивание DOCX.
3. Добавить пользователей и подписи.
4. Добавить опасности и меры безопасности.
5. Проверить длинные тексты и таблицы.
6. Только после этого переносить полный производственный шаблон.
