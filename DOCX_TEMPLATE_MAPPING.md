# DOCX_TEMPLATE_MAPPING.md

## Назначение

Документ фиксирует карту соответствия между переменными DOCX-шаблона и данными системы `permit-system` для наряда-допуска `Permit`.

Карта нужна для проверки реального DOCX-шаблона организации перед загрузкой в `DocumentTemplate` и генерацией через основной сервис `generate_permit_docx`.

> Важно: этот файл описывает соглашение по шаблонам и проверочный список. Он не меняет модели, workflow согласований или сервис генерации.

## Рекомендуемый подход

- Для новых шаблонов используйте **только безопасные транслит-переменные**: `{{ nomer_naryada }}`, `{{ mesto_rabot }}`, `{{ uchastniki_rabot }}`.
- Не используйте кириллицу внутри имён переменных DOCX-шаблона.
- Существующие английские переменные остаются совместимыми и не удаляются, чтобы старые шаблоны продолжали работать.
- Даты в транслит-переменных отдаются в формате `ДД.ММ.ГГГГ`.
- Списки опасностей и мер безопасности отдаются готовым текстом через запятую.
- Участники наряда отдаются готовым текстом: если участников несколько, каждый участник выводится с новой строки.
- Справочники и ручные поля используются совместно: справочники остаются для типовых значений, а ручные поля уточняют нестандартные случаи.
- Пустые значения, например отсутствие оборудования или дополнительных условий, отдаются как пустая строка.

## Почему переменные пишутся транслитом

DOCX-шаблоны обрабатываются через `docxtpl`/Jinja и затем открываются в Microsoft Word или LibreOffice. Кириллица внутри имён переменных может быть чувствительна к особенностям Word-разметки, автозамены, переносов и форматирования частей текста. Поэтому основная система переменных использует латиницу и подчёркивания: это стабильнее для Jinja, проще проверяется автоматически и меньше зависит от редактора DOCX.

Русский язык остаётся в описаниях, подписях и значениях, но не внутри имени переменной между фигурными скобками.

## Переменные участников наряда

| Переменная шаблона | Что означает | Источник данных | Пример значения | Комментарий |
| --- | --- | --- | --- | --- |
| `otvetstvennye_rukovoditeli` | Все ответственные руководители | `PermitParticipant.role = responsible_manager` | `Иванов И.И. — мастер` | Если несколько человек, каждый с новой строки. |
| `proizvoditeli_rabot` | Все производители работ | `PermitParticipant.role = work_producer` | `Петров П.П. — мастер` | Если участников нет, используется fallback из старого ручного поля. |
| `ispolniteli` | Исполнители работ | `PermitParticipant.role = performer` | `Сидоров С.С. — слесарь` | Выводится ФИО/должность или ручной ввод. |
| `chleny_brigady` | Члены бригады | `PermitParticipant.role = brigade_member` | `Кузнецов К.К. — электромонтёр` | Каждый участник с новой строки. |
| `dopuskayushchie` | Допускающие | `PermitParticipant.role = admitting_person` | `Допускающий по смене` | Может быть выбран из справочника или введён вручную. |
| `nablyudayushchie` | Наблюдающие | `PermitParticipant.role = observer` | `Наблюдающий участка` | Может быть пустой строкой. |
| `prochie_uchastniki` | Прочие участники | `PermitParticipant.role = other` | `Представитель подрядчика` | Для нестандартных ролей. |
| `uchastniki_rabot` | Все участники наряда | Все `PermitParticipant` данного Permit | `Иванов И.И. — мастер` | Общий список без группировки по ролям. |

## Справочник персонала и ручной ввод участника

- `PersonnelGroup` хранит группы работников: мастера, машинисты, слесари, электромонтёры, ИТР и другие категории.
- `Personnel` хранит работников из справочника. Эти записи **не являются пользователями сайта**, у них нет логина и пароля.
- `User` используется только для входа в систему, ролей operator/master/chief/admin и аудита действий.
- `PermitParticipant` связывает наряд с участниками. В каждой строке можно выбрать работника из справочника или указать ручное имя, если работника ещё нет в справочнике.
- Старые ручные поля `Permit.responsible_manager_text` и `Permit.work_producer_text` сохранены для совместимости и fallback-значений.

## Нерекомендуемые кириллические переменные

Кириллические имена переменных, использовавшиеся в ранних черновиках шаблонов, больше не являются основной системой. Для новых шаблонов не добавляйте кириллицу внутрь `{{ ... }}`. Если старый шаблон содержит такие имена, лучше заменить их на транслит-переменные из таблицы выше.

## Что нужно проверить на реальном шаблоне

Перед использованием реального DOCX-шаблона организации нужно проверить:

- [ ] Названия переменных совпадают с фактическим контекстом `generate_permit_docx`.
- [ ] В шаблоне нет опечаток в именах переменных.
- [ ] Переменные написаны транслитом, например `{{ nomer_naryada }}`.
- [ ] Имя переменной не разорвано переносом строки или отдельным форматированием Word.
- [ ] Таблицы DOCX корректно сохраняют структуру после генерации.
- [ ] Перенос строк в списках участников выглядит читаемо.
- [ ] Поля подписей расположены корректно и не съезжают после генерации.
- [ ] Даты отображаются в формате `ДД.ММ.ГГГГ`.
- [ ] Списки опасностей отображаются в согласованном формате.
- [ ] Списки мер безопасности отображаются в согласованном формате.
- [ ] Длинные тексты в `opisanie_rabot` не ломают таблицы и страницы.
- [ ] Форматирование после генерации соответствует исходному шаблону: шрифты, интервалы, границы таблиц, выравнивание.
- [ ] Пустое оборудование отображается корректно, если наряд создаётся без `Equipment`.
- [ ] Готовый DOCX открывается в Microsoft Word/LibreOffice без предупреждений о повреждении файла.

## Рекомендации по первому реальному шаблону

1. Начать с минимального шаблона, где есть только основные поля: `nomer_naryada`, `status_naryada`, `uchastok`, `vid_rabot`, `data_nachala`, `data_okonchaniya`, `mesto_rabot`.
2. Проверить генерацию и скачивание DOCX.
3. Добавить участников: `otvetstvennye_rukovoditeli`, `proizvoditeli_rabot`, `ispolniteli`, `uchastniki_rabot`.
4. Добавить опасности и меры безопасности.
5. Проверить длинные тексты и таблицы.
6. Только после этого переносить полный производственный шаблон.
